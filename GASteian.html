勤怠打刻システム（スマホアプリ版）アーキテクチャ及び開発提案書

本ドキュメントは、現在進めている「勤怠打刻システムのスマホアプリ化」における、システムアーキテクチャ、APIインターフェース設計、および外部API連携の要件を定義したものです。

1. システムアーキテクチャ図（フロントエンド 〜 GAS連携）

バックエンドにはGoogle Apps Script (GAS) のWeb App機能（doPost/doGet）を採用し、サーバーレスアーキテクチャで構築します。

1-1. 認証フロー（初回ログイン・サイレントログイン）

Google Sign-In SDKのサイレントログイン機能を利用するため、初回のみ画面でのログイン操作が必要ですが、2回目以降はバックグラウンドで自動的にトークンを更新します。そのため、毎回の再ログインは不要となり、従業員の打刻の手間を取らせません。

2. スマホアプリ側の実装スコープと技術要件

アプリ側はプレゼンテーション層に徹し、ステートを持たないThin Clientとして実装します。

コンポーネント 実装要件・留意点
認証機能 (OIDC) Google Sign-In SDK (または AppAuth) を用いてOpenID Connectフローを実装し、id_token (JWT) を取得・保持してください。
API通信 (HTTP Client) GASへのPOSTリクエストを実装します。GASの仕様によるCORSエラーを回避するため、リクエストヘッダーは Content-Type: text/plain などを利用するか、アプリ側のネイティブHTTPモジュールを利用してください。
レスポンスハンドリング 【重要】 GASは内部エラー発生時でもHTTP Status 200 OK を返す仕様があります。必ずレスポンスボディ（JSON）内の statusCode フィールドをパースしてエラーハンドリングを行ってください。
OAuth認可URLの処理 Slack/freeeの初回連携機能を利用する場合、APIから返却される認可URLを SFSafariViewController や Custom Tabs で開き、GASのコールバック(doGet)に遷移させる処理が必要です。

3. APIインターフェース仕様

RESTfulではなく、単一のエンドポイントに対して action パラメータでルーティングを行うRPCスタイルを採用しています。

エンドポイント

POST https://script.google.com/macros/s/{DEPLOY_ID}/exec

リクエストペイロード (JSON)

{
  "id_token": "eyJhbGciOiJSUzI1NiIs...", // 必須: Googleログインで取得したJWT
  "action": "clock_in",              // 必須: アクション指定 (clock_in/clock_out/day_off)
  "payload": {                       // 任意: 追加データが必要な場合
    "location": "本社",
    "device": "iOS"
  }
}

レスポンスペイロード (成功時)

{
  "statusCode": 200,
  "message": "出勤打刻を記録しました",
  "timestamp": "2023-10-25T09:00:00Z"
}

4. 外部APIアプリケーション作成と権限の申請

本番稼働に向けて、以下の各プラットフォームにおける「開発者アプリ」の作成許可及び設定作業をお願いいたします。

  • Google Cloud Console (GCP)
    • 目的: アプリ側で使用する「OAuth クライアント ID（Web/iOS/Android用）」の発行。
    • 要件: GAS側でのトークン検証（aud クレームの照合）にこのClient IDを使用し、他アプリからのなりすましを防止します。
    • セキュリティ補足（Client IDの漏洩リスク）: Client ID自体は公開情報に近いため、万が一漏洩しても直接的なデータ流出には繋がりません。しかし、悪意ある第三者がフィッシングアプリ等に流用するリスクがあるため、バックエンド（GAS）側での厳密な aud（宛先）検証を必須実装とし、不正利用を完全にブロックします。
  • Slack API (api.slack.com)
    • 目的: 打刻時のチャンネル自動通知。
    • 要件: User Token（ユーザー権限トークン）の発行。Botとしてではなく、特定ユーザーのアカウントを通じて打刻通知を行う構成とします。必要なスコープは chat:write 等となります。
  • freee Developer (app.secure.freee.co.jp)
    • 目的: freee人事労務へのタイムレコーダー情報書き込み。
    • 要件: コールバックURIとしてGASのデプロイURLを登録し、Client ID / Secretを取得します。

5. 【重要】freee API連携に関する運用アーキテクチャの提案

💡 提案事項： freeeのAPI認可（OAuthフロー）は「システム管理者アカウント」による一括管理とし、GASのScript PropertiesにRefresh Tokenを永続化する方式を推奨します。

背景と理由（技術的課題）

通常、ユーザー単位でfreeeのOAuth認可を通す設計にした場合、以下の問題が発生します。

  1. 運用コストの増大: freeeの仕様上、API経由で打刻を行うためには、事前にfreee管理画面から該当ユーザーの「タイムレコーダー機能」を個別にONにする設定が必要です。
  2. UXの低下: 全従業員にアプリ初回起動時に「freeeログイン＆アクセス許可」のOAuthフローを踏ませる必要があります。

解決策（管理者トークン方式）

GAS内部に「全権限を持つ管理者のfreee Access/Refresh Token」を保持させます。

  • GASは打刻リクエストを受け取った際、ユーザーの id_token に含まれるメールアドレスから、内部の社員マスタ（スプレッドシート等）を照会し、対象ユーザーの freee上での employee_id を特定します。
  • GASが管理者のトークンを用いて、POST /api/v1/time_clocks（該当 employee_id 宛）を叩きます。

既存業務フローとの親和性について

現在、事務の担当者様が管理者権限で打刻情報などの入力業務を行ってくださっていますが、本アーキテクチャはこの「管理者が一括してシステムに記録する」という現在の運用フローをそのままシステムに置き換える形となります。そのため、事務の方の現在の仕事内容や権限モデルを崩すことなく、極めてスムーズかつ安全に新しい打刻システムへ移行できるという大きなメリットがあります。

これにより、スマホアプリ開発側・従業員側の双方からfreee連携の複雑さを完全に隠蔽でき、保守性の高いアーキテクチャとなります。
